### 3. 飞书链路实测结论

- Bot 身份可私聊推送文本 + 图片  
- 发飞书前需**清除 HTTP 代理**，否则 token 易超时  
- `lark-cli --image` 必须使用 **cwd 相对路径**  
- **节点截图**（「微信二维码」img）优于整页截图，便于手机扫

### 生产模式定型（重要决策）

```text
# Linux / OpenClaw（默认生产）
PUBLISH_CHANNEL=api
PUBLISH_MODE=draft_notify_feishu
→ API 仅 draft → 飞书待办 → 管理员 mp 后台手点「发表」
→ 不默认 freepublish（规避可见性 bug）
→ 不默认 Linux 无头登录微信 / Xvfb 正发

# Windows 本机（保留完整能力，未破坏）
PUBLISH_CHANNEL=browser
PUBLISH_MODE=browser_full
→ Chrome DevTools 发表 + 可选飞书推验证码
```

### 两类飞书消息（避免混用）

## 三、关键工程结论

1. **freepublish 技术成功 ≠ 后台手动发表的运营可见性**；服务器日更应避免默认正发 API。  
2. **双 ProseMirror**：标题区与正文区不可写混。  
3. **封面**：正文插图后「从正文选择」最稳；可指定第 N 张。  
4. **代理三分**：生图可代理；微信 API、飞书 lark-cli 建议直连。  
5. **多账号**：顶栏显示名 + URL `token` 自检；作者跟当前号；归档用 `{slug}-*.json`。  
6. **Xvfb**：仅解决 Linux 无显示器渲染；**不能**替代登录态与管理员扫码；当前推荐路径不依赖它。  
7. **Windows 自动化路径完整保留**，仅调整服务器默认推荐。
